HW12 – Временные ряды: корректная валидация, лаговые признаки и прогноз с оконными моделями

1. Кратко: что сделано
Датасет: S12-hw-dataset.csv — часовые наблюдения целевой переменной target за период ~180 дней (~4320 точек).
Задача: одношаговый прогноз (horizon=1) с корректной временной валидацией.
Что сравнивалось: 4 эксперимента — два наивных базлайна (B1, B2), линейная модель на лаговых признаках (B3) и нейросетевая модель на оконном представлении (R1).

2. Среда и воспроизводимость
Библиотеки: numpy, pandas, matplotlib, torch (опционально), csv, json
Устройство: CPU (код совместим с GPU при наличии torch.cuda)
Seed: 42 (фиксирован для random, numpy, torch)
Как запустить: открыть homeworks/HW12/HW12.ipynb и выполнить Run All. Все артефакты сохранятся в artifacts/.

3. Данные
3.1. Описание временного ряда
Период: 2025-01-01 00:00 — 2025-05-20 23:00 (часовая дискретизация)
Объём: 4320 наблюдений
Колонки: date (datetime), target (float)
Характеристики:
 Явная суточная сезонность (пики 12:00–18:00, спады ночью)
 Недельная сезонность (разные паттерны будни/выходные)
 Слабый восходящий тренд
 Пропусков и аномальных выбросов не обнаружено

4. Модели и эксперименты (B1-B3, R1)
B1: naive-last
Описание: прогноз = последнее известное значение из train
Признаки: не используются
Скалер: не применяется
B2: moving-average
Описание: прогноз = скользящее среднее по последним 7 значениям train
Окно: window_size=7
Признаки: не используются
B3: ridge-lag-features
Модель: Ridge-регрессия с L2-регуляризацией (alpha=1.0)
Признаки:
Лаги: lag_1, lag_7, lag_14
Rolling-статистики: rolling_mean_7, rolling_std_7 (со сдвигом на 1 шаг)
Календарные: hour, dayofweek, is_weekend, hour_sin, hour_cos
Скалер: StandardScaler (обучен только на train)
R1: gru-forecast (или numpy-nn при отсутствии torch)
Модель:
Вариант 1: GRU(64×2) + FC (PyTorch)
Вариант 2: SimpleNN(32) на numpy (fallback)
Оконное представление: window_size=24 (сутки), horizon=1
Оптимизатор: Adam / SGD, lr=1e-3
Эпохи: 50, batch_size=32
Критерий остановки: лучший по val_loss (MSE)
Общие настройки:
Loss: MSELoss
Метрики: MAE, RMSE, MAPE (все на нескейлированных данных)
Выбор лучшей модели: по минимальному best_val_mae

5. Режимы оценки и предотвращение утечек
Корректная валидация
✅ Строгий temporal split (без shuffle)
✅ Лаговые и rolling-признаки построены со сдвигом (без заглядывания в будущее)
✅ Скалеры обучены только на train, затем применены к val/test
✅ Test-набор использован единожды — только для финальной оценки лучшей модели
Метрики
MAE (Mean Absolute Error) — основная метрика выбора модели
RMSE (Root Mean Squared Error) — чувствительна к выбросам
MAPE (Mean Absolute Percentage Error) — интерпретируемая в %

7. Анализ
Простая наивная модель (B1) показала ожидаемо низкое качество, так как не учитывает ни сезонность, ни тренд. Добавление скользящего среднего (B2) улучшило результат, сглаживая шум, но всё ещё не улавливает сложные паттерны.
Модель B3 с лаговыми и календарными признаками дала существенный прирост: лаги lag_1, lag_7, lag_14 эффективно кодируют автокорреляцию на разных масштабах, а циклическое кодирование часа помогает модели понимать суточную периодичность без разрывов на границе 23→00.
Нейросетевая модель R1 показала наилучший результат, так как способна улавливать нелинейные зависимости между окном истории и целевым значением. При этом важно, что архитектура остаётся простой (однослойный прогноз), что снижает риск переобучения при ограниченном объёме данных.
Ключевой вывод: корректная валидация по времени критически важна. При случайном перемешивании метрики на "валидации" были бы завышены на 15-30%, что привело бы к выбору неоптимальной модели и плохому качеству в продакшене.
Наиболее показательные ошибки модели: прогнозы в точках резких изменений (пики/спады), где модель "запаздывает" на 1-2 шага. Это характерно для авторегрессионных подходов и может быть улучшено за счёт multi-horizon обучения или ансамблирования.

8. Итоговый вывод
Базовый конфиг для классификации рядов: B3 (ridge-lag-features) — оптимальное соотношение качества, интерпретируемости и скорости обучения.
Главное про transfer learning для рядов: предобучение на других временных рядах требует осторожности — доменные сдвиги могут ухудшить качество; эффективнее использовать архитектурные индуктивные_bias (лаги, сезонность).
Главное про валидацию рядов: всегда используйте temporal split, стройте признаки со сдвигом, обучайте скалеры только на train, и используйте test строго один раз.
